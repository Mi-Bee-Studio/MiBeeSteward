# MiBee Steward 介绍

## 什么是 MiBee Steward

MiBee Steward 是一个**设备/网络层的资产发现、识别与登记**工具 —— 面向网络与 IoT 资产的轻量 CMDB。它自动发现网络上有哪些设备（通过 ICMP、TCP 端口扫描、SNMP、HTTP、RTSP、ONVIF、mDNS、WS-Discovery 探测，外加可选的 eBPF 被动观测），推断它们是什么（通过协议指纹识别设备类型、品牌、型号），并基于心跳鲜活度持续登记/跟踪。系统采用 Go 后端（Chi + SQLite，CGO-free）+ 通过 go:embed 嵌入的 SvelteKit 单页应用，以单个零依赖二进制交付。

心跳探测保持资产登记的鲜活度：设备以可配置间隔被探测，连续三次失败后自动标记为离线（设备再次响应时自动恢复）。这产生了在线/离线历史、延迟和可用性统计，使资产登记成为持续可信的活账本，而非一次性快照。资产状态也通过 `/metrics` 和 `/sd` 暴露给 Prometheus 生态 —— **告警与可视化有意留给 Alertmanager 和 Grafana**，它们原生消费这些端点。详见[产品范围与边界](product-scope.md)。

## 核心功能

### 设备管理
- 网络设备的完整 CRUD 操作，包括创建、编辑、删除和查看
- 设备分组和分类，便于有序管理
- 设备系统管理，支持为每个设备添加多个安装的系统，包含入口 URL
- 详细的设备配置，支持自定义参数和探测设置
- 批量操作，高效管理大型设备清单
### 多协议探测
- SNMP v2c 支持，使用 community string 认证识别网络设备
- ICMP ping 功能，用于基础网络连接测试
- TCP 连接检查，用于特定端口可用性和服务识别
- HTTP GET 请求，支持自定义标头和超时设置
- 指数退避重试机制（1s → 2s → 4s），仅适用于网络错误

### 心跳（资产鲜活度）
- 每个设备或设备组可配置的探测间隔（保持资产登记鲜活）
- 三次失败检测，自动更新状态
- 失败设备再次响应时的自动恢复检测
- 实时状态变化反映在网页界面和指标中

### Prometheus 集成
- `/metrics` 处的完整指标端点，遵循 Prometheus 导出格式
- 显示在线/离线状态的设备状态指标
- 设备系统自动发现，支持 `metrics_enabled=true` 的系统在 `/sd` 端点自动发现
- 用于性能监控的心跳计数器（总尝试次数和失败尝试次数）
- 响应时间直方图
- `/sd` 处的 HTTP 服务发现端点，支持自动发现
### 嵌入式 Web 界面
- 通过 go:embed 嵌入的 SvelteKit 5 单页应用
- 使用 ECharts 的实时仪表板，用于数据可视化
- 深色和浅色主题支持，自动检测系统偏好
- 为桌面和移动设备优化的响应式设计
- 设备证书和配置文件的文件上传功能

### 认证与安全
- 基于 JWT 的认证，采用 cookie 优先方式
- API 客户端和自动化工具的 Bearer 令牌回退
- 基于角色的访问控制，管理员（完全访问）和用户（只读）角色
- 所有用户操作的全面审计日志，包括登录尝试、设备变更和文件上传
- CSRF 保护和安全标头中间件

### 国际化
- 完整的 i18n 支持，提供英语和中文语言包
- 使用 @inlang/paraglide-js 进行高效的消息编译
- 基于浏览器偏好的自动语言检测
- 本地化的日期/时间格式和错误消息

## 使用场景

### 网络资产清单
自动发现并登记网络上的设备 —— 路由器、交换机、无线 AP、服务器、IoT —— 通过协议指纹识别品牌/型号。用持续鲜活的资产登记替代基于电子表格的手工资产跟踪。

### IoT / 摄像头舰队发现
按品牌和型号识别 IP 摄像头、传感器、控制器及其他 IoT 设备。摄像头检测（RTSP + ONVIF + camera 分类器）是当前的优先场景，因为指纹清晰、需求明确 —— 不代表 MiBee Steward 是摄像头专属工具。同一套识别管线适用于任何设备类型。

### 分支机构 / SOHO 网络画像
在 LibreNMS 或 Zabbix 显得过重的小型/分支网络中足够轻量。部署单个二进制、扫描子网、得到结构化的资产画像，无需搭建数据库、消息代理或容器栈。

### 实验室 / 边缘资产跟踪
用灵活的逐设备探测配置跟踪研究实验室设备、测试装置、测量设备和边缘节点。基于心跳的鲜活度确保资产登记反映现实，而非过期快照。

## 系统要求

### 后端要求
- Go 1.26 或更高版本，用于编译和运行
- 通过 modernc.org/sqlite 的 SQLite（CGO_ENABLED=0 构建标志）
- 应用程序和数据库至少需要 50MB 磁盘空间
- 网络访问用于设备探测和指标收集
- 可选：文件上传和文档存储需要 100MB+ 空间

### 前端要求
- 开发和构建需要 Node.js 20+
- 具有 JavaScript 支持的现代网络浏览器
- 可选：ECharts 用于仪表板可视化功能

### 基础设施要求
- Linux x86_64（主要部署平台）
- 可选：通过交叉编译支持 ARM64
- 生产服务管理使用 Systemd
- Nginx 用于反向代理和 TLS 终止
- Docker 用于容器化部署

### 性能考虑
- WAL 模式的单一 SQLite 数据库，支持并发访问
- 大多数部署的内存使用通常低于 100MB
- CPU 使用量随活动设备探测数量而扩展
- 网络带宽取决于探测频率和响应大小

## 架构

```
Browser → Nginx → Go Server (Chi)
                         ↓
               SQLite + Heartbeat Service (10s ticker)
                         ↓
               Prometheus Metrics / HTTP Service Discovery
```
