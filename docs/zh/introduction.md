# MiBee Steward 介绍

## 什么是 MiBee Steward

MiBee Steward 是为网络管理员和 DevOps 团队设计的全面设备管理与监控系统。系统采用 Go 后端构建，使用 Chi Web 框架和 SQLite 数据库，并包含一个通过 go:embed 嵌入的 SvelteKit 单页应用，提供现代化的网页界面。系统支持跨多种协议的实时监控能力，包括 SNMP v2c、ICMP ping、TCP 连接检查和 HTTP GET 请求。其单二进制部署架构使其能够在各种环境中轻松部署和管理。

系统基于心跳监控原理运行，设备以可配置的时间间隔进行探测。如果设备在连续三次尝试后仍未响应，其状态将自动更新以表示停机。这种方法提供可靠、自动化的监控，无需人工干预。整个应用程序可以作为一个可执行文件和配置文件进行部署，简化了分发和维护。

## 核心功能

### 设备管理
- 网络设备的完整 CRUD 操作，包括创建、编辑、删除和查看
- 设备分组和分类，便于有序管理
- 设备系统管理，支持为每个设备添加多个安装的系统，包含入口 URL
- 详细的设备配置，支持自定义参数和探测设置
- 批量操作，高效管理大型设备清单
### 多协议探测
- SNMP v2c 支持，使用 community string 认证监控网络设备
- ICMP ping 功能，用于基础网络连接测试
- TCP 连接检查，用于特定端口可用性和服务监控
- HTTP GET 请求，支持自定义标头和超时设置
- 指数退避重试机制（1s → 2s → 4s），仅适用于网络错误

### 心跳监控
- 每个设备或设备组可配置的监控间隔
- 三次失败检测，自动更新状态
- 失败设备再次响应时的自动恢复检测
- 实时状态变化反映在网页界面和指标中

### Prometheus 集成
- `/metrics` 处的完整指标端点，遵循 Prometheus 导出格式
- 显示在线/离线状态的设备状态指标
- 设备系统自动发现，支持 `metrics_enabled=true` 的系统在 `/sd` 端点自动发现
- 用于性能监控的心跳计数器（总尝试次数和失败尝试次数）
- 响应时间直方图
- `/sd` 处的 HTTP 服务发现端点，支持自动发现
### 嵌入式 Web 界面
- 通过 go:embed 嵌入的 SvelteKit 5 单页应用
- 使用 ECharts 的实时仪表板，用于数据可视化
- 深色和浅色主题支持，自动检测系统偏好
- 为桌面和移动设备优化的响应式设计
- 设备证书和配置文件的文件上传功能

### 认证与安全
- 基于 JWT 的认证，采用 cookie 优先方式
- API 客户端和自动化工具的 Bearer 令牌回退
- 基于角色的访问控制，管理员（完全访问）和用户（只读）角色
- 所有用户操作的全面审计日志，包括登录尝试、设备变更和文件上传
- CSRF 保护和安全标头中间件

### 国际化
- 完整的 i18n 支持，提供英语和中文语言包
- 使用 @inlang/paraglide-js 进行高效的消息编译
- 基于浏览器偏好的自动语言检测
- 本地化的日期/时间格式和错误消息

## 使用场景

### 网络监控
MiBee Steward 擅长监控网络基础设施，包括路由器、交换机和防火墙。IT 管理员可以跟踪设备可用性、响应时间和服务可用性，监控整个网络。SNMP 探测功能允许监控网络统计信息、接口计数器和系统健康指标，提供全面的网络可见性。

### IoT 设备管理
非常适合管理物联网设备，如智能传感器、摄像头和工业控制器。系统支持针对 IoT 设定的自定义探测间隔和特定协议，允许通过 HTTP 端点监控专有协议。设备分组有助于按位置、功能或重要性组织大量 IoT 设备。

### 服务器机房监控
适用于监控数据中心的服务器可用性和服务健康状况。TCP 连接检查确保关键服务正在运行，而 HTTP GET 探测验证 Web 应用程序正确响应。系统可以监控多个可用区域的服务器，并在性能问题影响用户之前提供早期警告。

### 实验室设备跟踪
对于管理研究实验室设备、测试装置和测量设备至关重要。灵活的探测配置允许监控具有特定协议和间隔的自定义设备。自动故障检测确保及时识别昂贵的实验室设备问题，最大限度地减少停机和研究延迟。

## 系统要求

### 后端要求
- Go 1.26 或更高版本，用于编译和运行
- 通过 modernc.org/sqlite 的 SQLite（CGO_ENABLED=0 构建标志）
- 应用程序和数据库至少需要 50MB 磁盘空间
- 网络访问用于设备探测和指标收集
- 可选：文件上传和文档存储需要 100MB+ 空间

### 前端要求
- 开发和构建需要 Node.js 20+
- 具有 JavaScript 支持的现代网络浏览器
- 可选：ECharts 用于仪表板可视化功能

### 基础设施要求
- Linux x86_64（主要部署平台）
- 可选：通过交叉编译支持 ARM64
- 生产服务管理使用 Systemd
- Nginx 用于反向代理和 TLS 终止
- Docker 用于容器化部署

### 性能考虑
- WAL 模式的单一 SQLite 数据库，支持并发访问
- 大多数部署的内存使用通常低于 100MB
- CPU 使用量随活动设备探测数量而扩展
- 网络带宽取决于探测频率和响应大小

## 架构

```
Browser → Nginx → Go Server (Chi)
                         ↓
               SQLite + Heartbeat Service (10s ticker)
                         ↓
               Prometheus Metrics / HTTP Service Discovery
```
