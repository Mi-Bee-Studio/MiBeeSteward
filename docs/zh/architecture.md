# 架构

## 系统概述

MiBee Steward 是一个**设备/网络层的资产发现、识别与登记**工具，构建为具有嵌入式 Web 界面的单二进制部署。系统由使用 Chi Web 框架和 SQLite 数据库的 Go 后端，以及通过 Go 的 `go:embed` 功能嵌入的 SvelteKit 5 前端组成。系统通过 SNMP/ICMP/TCP/HTTP/RTSP/ONVIF 等协议自动发现设备并推断其身份，通过心跳保持资产登记鲜活，并通过 `/metrics` 与 `/sd` 将资产状态暴露给 Prometheus 生态（告警与可视化留给 Alertmanager/Grafana）。详见[产品范围与边界](product-scope.md)。

## 分层架构

系统遵循领域驱动设计 (DDD) 启发的分层架构，采用可选的仓储模式：

```
┌─────────────────────────────────────────────────────────────┐
│                    前端层                                  │
│                 SvelteKit SPA（嵌入式）                   │
└─────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐
│                   处理器层                                 │
│              HTTP 请求/响应处理                            │
└─────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐
│                   服务层                                  │
│              业务逻辑与编排                               │
└─────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐
│              仓储层（可选）                                │
│             数据访问抽象层                                │
└─────────────────────────────────────────────────────────────┘
┌─────────────────────────────────────────────────────────────┐
│                    领域层                                  │
│               DTO、常量、上下文键                          │
└─────────────────────────────────────────────────────────────┘
```

### 层职责

**领域层**：包含业务模型、数据传输对象 (DTO) 和应用程序常量。包括用于请求跟踪的上下文键和类型化错误定义。

**仓储层**：可选的数据访问层，封装 sqlc 生成的代码。由 DeviceService 使用，而其他服务直接使用 sqlc 进行更简单的数据访问模式。

**服务层**：实现业务逻辑、错误处理和探测子系统。服务使用基于构造函数的依赖注入并返回类型化错误。

**处理器层**：HTTP 请求/响应处理、输入验证、审计日志记录，以及将错误转换为 HTTP 状态码。

## 请求生命周期

HTTP 请求通过以下流程在系统中流转：

1. **HTTP 请求** → 外部客户端向服务器发送请求
2. **Chi 路由器** → 路由匹配和中间件链启动
3. **中间件管道** → 每个中间件按顺序处理请求
4. **处理器** → 请求验证、审计日志记录、业务逻辑调用
5. **服务** → 业务逻辑处理和错误处理
6. **仓储/SQLc** → 对 SQLite 数据库的数据访问
7. **数据库响应** → 查询结果返回给服务
8. **服务响应** → 业务逻辑结果返回给处理器
9. **HTTP 响应** → 最终响应格式化并发送给客户端

## 中间件链

系统通过精心排序的中间件链处理请求：

```
请求 → 请求ID → 真实IP → 日志 → 指标 → 恢复器 → 
安全头 → CORS → CSRF → 速率限制 → 认证/RBAC → 处理器
```

每个中间件都有特定用途：
- **请求ID**：分配唯一的请求 ID 用于跟踪
- **真实IP**：从代理中提取真实客户端 IP
- **日志**：使用结构化数据进行请求/响应日志记录
- **指标**：Prometheus 指标收集
- **恢复器**：恐慌恢复和错误处理
- **安全头**：安全相关的 HTTP 头
- **CORS**：跨域资源共享配置
- **CSRF**：跨站请求伪造保护
- **速率限制**：请求速率限制
- **认证/RBAC**：基于角色的访问控制

## 数据库设计

系统使用带有预写日志 (WAL) 模式的 SQLite 以获得最佳性能：

- **连接池**：单个连接 (`MaxOpenConns=1`) 以兼容 SQLite
- **代码生成**：sqlc 从 SQL 查询生成类型安全的 Go 代码
- **迁移系统**：启动时自动执行嵌入的 schema.sql
- **数据目录**：默认路径 `./data/mibee.db`，自动创建目录
- **模式演进**：迁移处理数据库模式更改

数据库代码生成遵循以下模式：
```
db/queries/*.sql → sqlc generate → internal/db/*.go
```

切勿直接编辑 `internal/db/*.go` 文件 - 修改 SQL 查询并重新生成。

## 前端架构

前端是具有特定配置的 SvelteKit 5 单页应用程序：

- **路由**：`web/src/routes/` 中的基于文件的路由
- **构建输出**：前端构建到 `web/dist/` 用于嵌入
- **嵌入**：Go 通过 `web/embed.go` (`//go:embed all:dist`) 嵌入前端
- **样式**：Tailwind 4 用于响应式设计
- **图表**：ECharts 集成用于数据可视化
- **国际化**：@inlang/paraglide-js 支持英语和中文
- **状态管理**：SvelteKit 的 `$app/stores` 用于客户端状态

关键前端模式：
- 翻译函数：`t('section', 'key')`
- 带有 ECharts 集成的图表组件
- 带有 `+page.svelte` 文件的基于文件的路由
- `web/src/lib/components/` 中的共享组件

## 后台服务

系统在单独的 goroutine 中运行后台服务：

**HeartbeatService**：
- 10 秒计时器间隔
- 基于监控间隔的可配置 `isDue()` 检查
- 3 次失败阈值用于设备检测
- 协调所有设备上的探测执行
- 通过返回的服务引用进行优雅关闭协调

**服务生命周期**：
```
NewRouter() → HeartbeatService.Start() → goroutine 启动 → 信号等待 → 优雅关闭
```

## 探测子系统

探测系统使用基于接口的设计和装饰器模式：

```
Prober 接口 → ICMP/TCP/HTTP/SNMP 实现 → RetryProber 装饰器
```

**探测功能**：
- 指数退避重试（1秒 → 2秒 → 4秒）
#HW>- 仅网络错误重试（探测失败时立即返回）
#TR>- 可配置的超时和重试参数
#YH>- 全面的探测结果跟踪

**探测实现**：
- **ICMP**：基于 ping 的连通性检查
- **TCP**：端口连通性和横幅抓取
- **HTTP**： Web 服务器健康检查和响应验证
- **SNMP**：通过 SNMP 协议进行网络设备监控

## 网络扫描器（v2）

扫描器采用**插件式、五层架构**，将检测与持久化解耦，新增协议只需注册一个分类器 + 一个处理器，无需改动编排或持久化层。它取代了旧的硬编码六阶段流水线。

```
┌─────────────────────────────────────────────────────────────────────┐
│  ① 探测 (Probe)    — 主动（TCP/SNMP/RTSP/ONVIF/HTTP-metrics）+       │
│                      被动（eBPF TC 观测器，构建标签隔离）             │
│     → 产出 Evidence（port_open / banner / snmp / wsdiscovery / ...） │
├─────────────────────────────────────────────────────────────────────┤
│  ② 分类 (Classifier) — 每协议一个，对 Evidence 的纯函数              │
│     → 融合成 ServiceIdentity（ssh/http/rtsp/onvif/prometheus/        │
│       node_exporter/snmp/camera），带置信度                          │
├─────────────────────────────────────────────────────────────────────┤
│  ③ 服务处理器 (ServiceHandler) — 按服务定制                          │
│     ├─ GenerateHeartbeat()  → 生成适配的心跳配置                     │
│     ├─ Collect()            → 深度采集，可 Trigger 触发其他处理器    │
│     └─ EnrichDevice()       → 用采集结果补充设备字段                 │
│     典型级联：http → 探测 /metrics → prometheus → 发现 node_ →        │
│     node_exporter → 解析 CPU/内存/内核 → enrich                      │
├─────────────────────────────────────────────────────────────────────┤
│  ④ 持久化 (Persistence) — Repository 接口（SQLite 实现在 store/）    │
│     → 记录 evidence / services / 设备更新 / 心跳                     │
├─────────────────────────────────────────────────────────────────────┤
│  ⑤ 编排 (Orchestrator) — 声明式驱动：gather → classify → dispatch   │
│     带环保护的级联触发（最大深度 5）                                  │
└─────────────────────────────────────────────────────────────────────┘
```

**扩展检测能力**：新增协议只需实现一个 `ServiceClassifier` 和一个 `ServiceHandler`，然后在 `classify.DefaultClassifiers()` / `handler.DefaultHandlers()` 中注册。编排层和持久化层无需改动。

**开箱即用的服务识别**：SSH、HTTP/HTTPS、RTSP、ONVIF、SNMP、Prometheus、node_exporter，以及主机级的 **camera** 元身份（由 RTSP + ONVIF 证据融合得出，支持从 Server 头 / SNMP sysDescr / 企业 OID 推断品牌）。

**eBPF 被动观测器**：可选的 TC ingress 程序（`bpf/tc_ingress.c`），嗅探 ONVIF WS-Discovery 多播（239.255.255.250:3702）和 TCP 魔术字节（SSH-/RTSP/HTTP/）。它产出置信度 0.6 的辅助 Evidence，与主动探测结果融合。仅通过 `make build-with-ebpf` 编译（需要 clang/llvm/bpftool + 内核 BTF ≥5.8）；默认构建使用 no-op stub，二进制无内核依赖。

**持久化表**（v2 新增）：`service_evidence`（原始探测观测，受采样控制）和 `host_services`（每主机的已分类服务身份）。设备 upsert 写入既有 `devices` 表；心跳配置写入 `heartbeat_configs`。

## Prometheus 集成

系统提供全面的监控功能：

**指标端点**：`/metrics` - 标准 Prometheus 指标导出
**仪表板代理**：`/api/v1/dashboard/query` - 对 Prometheus 的只读代理
**服务发现**：`/sd` 处的 HTTP SD 端点用于服务发现
**指标类型**：各种系统指标的计数器、仪表盘和直方图

仪表板服务配置为使用 Prometheus 后端进行监控数据查询。
